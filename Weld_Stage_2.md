# 05\. Alternative Agent Routes

Agents should not be locked into a single brittle implementation path\. The best route depends on the current codebase, time budget, and whether the team wants a safe MVP, the full target experience, or a cinematic stretch\. These routes are designed to be mutually understandable: a conservative route can later evolve into the target route without throwing work away\.

__Route__

__When to choose it__

__What the agent builds__

__Risks__

__Exit criteria__

A\. SyntaxBlock MVP

Need fast improvement with limited refactor\.

Build reusable SyntaxBlock and replace How It Works terminal cards\.

Hero and role text may still feel generic\.

LuaU code, line numbers, tabs, Stage 1 token colors are visible\.

B\. Full Studio surfaces

Target route for Stage 2\.

Build OutputPanel, SyntaxEditor, Explorer typography, Properties value typography, command input\.

Requires more CSS and JSX changes\.

All text surfaces feel like Studio tools and update from shared state\.

C\. Stateful teaching route

Want the page to explain product through code\.

Make script tabs drive explanations, logs, proof comments, and CTA readiness\.

Can overcomplicate if copy is weak\.

User can switch scripts and see product logic explained dynamically\.

D\. Accessibility\-first route

Current page has readability or keyboard issues\.

Implement text semantics, labels, aria\-live policy, focus\-visible, code as selectable text\.

Less cinematic in first pass\.

Keyboard and screen\-reader users get the same information and state\.

## Route selection heuristic

- If the codebase is fragile, start with the safe route and add dynamic contracts behind feature flags\.
- If the page already has clean component boundaries, use the target route and wire state immediately\.
- If the goal is a screenshot\-worthy launch, use the cinematic route after the target route passes accessibility and performance gates\.
- If mobile or performance is currently weak, choose the constraint\-reduced route before adding ambient motion or extra motifs\.

# 08\. Accessibility, Performance, and Brand\-Safety Gates

- Code blocks must be real text, not images\.
- Tabs must be real buttons or implement a full tablist pattern\.
- Explorer tree should use buttons with aria\-expanded unless the full ARIA tree keyboard model is implemented\.
- Log updates should be aria\-live polite only for meaningful state changes\.
- Line numbers should not be the only way to understand the code sample\.
- Do not reduce all text to tiny mono\. Use mono for code, values, and logs; use sans for explanation\.
